# 像设计师一样使用 Agent

> 索引：[[00-Agent 协作（索引）]]  
> 后续：[[02-像项目主管一样使用 Agent]] ｜ 汇总：[[03-两种 Agent 使用模式的协作与切换]]

## 0. 核心判断

开发者在使用 Agent 时，不应该只把自己放在“项目主管”的位置上：提出需求、等待产出、最后验收结果。更合适的姿态是：**像设计师一样参与系统的形成过程**。

这里的“设计师”不是 UI 设计师，而是系统语义、边界、节奏和验证方式的设计者。

## 1. 问题：Agent 太能干，反而会让我失去参与感

在传统开发中，开发者通常亲自经历从问题定义、模块拆分、接口设计、实现、调试到验证的全过程。这个过程虽然慢，但会让开发者逐渐形成对项目的理解。

但在 Agent 工作流中，很容易发生这样的情况：

```plaintext
你：做一个 xxx。
agent：好的，我已经实现了 A/B/C/D/E/F，还顺便重构了目录、加了测试、改了配置。
你：？？？我没参与，我也审不动。
```

这不是 Agent 不够强，而是协作协议出了问题。

当我只给 Agent 一个大目标时，它会倾向于一次性最大化完成目标：多写功能、多补边界、多加工程化配置、多做它认为“应该做”的事情。结果是项目确实变得“完整”了，但我对项目的理解却被跳过了。

于是，我从开发者退化成了验收者；从系统设计者退化成了代码审查员；从项目参与者退化成了等待 Agent 施工结束的人。

这不是我想要的开发方式。

## 2. 角色定位：不是 autonomous coder，而是 co-design implementer

很多时候，我并不需要一个完全自治的 Agent。至少在需要长期维护、需要形成工程判断、需要训练设计能力的项目里，我更需要的是一个共同设计的实现者：

```plaintext
不是：我给出目标，Agent 自己完成整个项目。
而是：我定义边界，Agent 在边界内完成下一个可验证步骤。
```

Agent 可以写代码、补测试、生成文档、整理目录，但它不应该在没有确认的情况下替我决定项目的形态。

我的角色不是“产品经理式地发需求”，而是：

- 定义系统目标；
- 定义最小闭环；
- 定义哪些事情暂时不做；
- 定义模块边界；
- 定义状态变化；
- 定义验证方式；
- 判断当前抽象是否过早、过重或过窄。

换句话说，我不应该只管理 Agent 的产出，而应该设计 Agent 的工作边界。

## 3. 工作机制：四把锁

为了避免 Agent 一次性吞掉整个项目，我需要在协作开始前给它加上四把锁。

### 3.1 Scope Lock：范围锁

一次只允许完成一个最小闭环，不允许提前实现未确认的 feature。

好的任务描述不是：

```plaintext
帮我做一个完整的任务管理系统。
```

而是：

```plaintext
只实现第一个 checkpoint：
用户可以创建一个 Task，系统保存它，并能查询它的状态。
暂时不要实现标签、搜索、权限、同步和 UI 美化。
```

范围锁的目标是防止 Agent 把“当前步骤”扩张成“完整项目”。

### 3.2 Design Lock：设计锁

在我确认设计前，不要写代码。

Agent 写代码之前，必须先给出本轮方案，包括：

- 本轮目标；
- 本轮不做什么；
- 会修改哪些文件；
- 为什么这样设计；
- 完成后如何运行和验证。

设计锁的目标是把我重新拉回项目中心。只有当我理解并确认这一步之后，Agent 才能进入实现。

### 3.3 Diff Lock：变更锁

每轮最多修改少量文件。默认可以限制为 3 个文件。

如果必须超过这个限制，Agent 需要先解释原因，并等待确认。

变更锁的目标不是限制 Agent 的能力，而是保护我的审查能力。一次性修改十几个文件，哪怕代码是对的，也会让我失去理解项目演化的机会。

### 3.4 Checkpoint Lock：检查点锁

每一步都必须可运行、可验证、可停下。

每轮结束后，Agent 必须回答：

1. 做到了什么；
2. 如何运行验证；
3. 下一步建议；
4. 当前未解决问题。

检查点锁的目标是让项目始终处在可接管状态，而不是被 Agent 推到一个我看不懂、也不敢改的地方。

## 4. 推荐开发节奏

我更适合使用一种“薄片式”的开发节奏，而不是一开始列出庞大的 feature list。

```plaintext
0. Vision：我想做什么
1. Slice：这一轮最小闭环是什么
2. Boundary：这一轮不做什么
3. Model：核心概念和状态是什么
4. Interface：用户 / 调用方如何使用
5. Implementation：Agent 写代码
6. Verification：跑起来、测试、看日志
7. Reflection：这个边界是否合理
8. Next Slice：进入下一片
```

其中第 7 步非常重要。

我不是只验收代码，而是验收设计：

- 这个抽象舒服吗？
- 这个边界会不会太早？
- 这个接口以后会不会绑死？
- 这个状态模型能不能解释真实情况？
- 这一轮实现之后，项目有没有变得更可理解？

如果缺少 Reflection，Agent 工作流就会变成连续施工；如果保留 Reflection，我才仍然是系统的设计者。

## 5. 工作卡片：设计卡片

为了稳定协作节奏，可以要求 Agent 每次先输出一张“设计卡片”，并且在设计卡片确认前不要写代码。

```markdown
请用设计卡片格式回答，不要写代码：

# 本轮目标

# 用户路径

# 核心对象

# 状态变化

# 文件影响

# 不做什么

# 风险

# 验证方式

# 等待我确认的问题
```

这张卡片的价值是：它让代码实现之前先出现一个可讨论的对象。

我可以修改目标、砍掉范围、调整状态机、改变文件边界，然后再让 Agent 实现。这样我参与的是设计，而不只是事后审查代码。

## 6. 参考 Prompt

### 6.1 项目开始时使用

```markdown
你不是 autonomous coder，而是我的 co-design implementer。

我们的目标不是一次性完成整个项目，而是一步步构建一个边界清晰、可运行、可审核的项目。

工作规则：

1. 不要一次性实现所有 features。
2. 每轮只完成一个最小可验证步骤。
3. 写代码前，必须先给出本轮方案，包括：
   - 本轮目标
   - 不做什么
   - 涉及哪些文件
   - 为什么这样设计
   - 完成后如何验证
4. 在我确认前，不要写代码。
5. 每轮最多修改 3 个文件；如果必须超过，先说明原因并等待确认。
6. 不要擅自引入复杂框架、重构目录、添加额外功能。
7. 每次实现后，只汇报：
   - 修改了哪些文件
   - 核心 diff 摘要
   - 如何运行
   - 如何验证
   - 下一步建议
8. 如果发现设计不清楚，先问问题，不要自行脑补。
9. 优先保持项目可运行，而不是追求完整。
10. 当前阶段只追求 walking skeleton，不追求 production-ready。

现在请先不要写代码。
请先帮我定义这个项目的：
- 核心目标
- 最小闭环
- 第一阶段不做的事情
- 初始模块边界
- 第一个可运行 checkpoint
```

### 6.2 确认设计后进入实现

```markdown
可以进入实现。

只实现刚才确认的 checkpoint。
不要实现下一阶段功能。
不要重构无关代码。
完成后给我运行方式和验证方式。
```

### 6.3 Agent 开始发散时打断

```markdown
停止扩展。

回到当前 checkpoint。
删除或撤回未确认的功能。
只保留当前最小闭环需要的改动。
```

### 6.4 改动过大、无法审查时收束

```markdown
这次改动太大，我无法审核。

请不要继续写新代码。
把当前改动拆成 3 个独立 commit 级别的步骤：

1. 基础结构
2. 核心逻辑
3. 验证与测试

每一步说明涉及文件、设计理由、验证方式。
```

## 7. 与其他模式的关系

“像设计师一样使用 Agent”并不意味着每个项目都必须慢慢推进。

有时我确实需要快速模式：让 Agent 在一个 session 中自我推进，快速搭建一个完整 demo，用来验证想法是否成立。这种模式是合理的，但它和共同设计模式不能混用。

可以这样区分：

```plaintext
Explorer Mode：快速生成一个可运行标本，用来观察项目形态。
Designer Mode：小步推进，严格控制边界，用来构建长期项目。
Review Mode：停止新增功能，审查已有代码，重新接管设计权。
Harden Mode：补测试、日志、错误处理、文档和工程化能力。
```

快速模式的产物不应该被直接视为最终架构。它更像一个工程标本：先让项目形态暴露出来，再由我解剖、审查、重构和收敛。

因此，关键不是“快”或“慢”，而是每次 session 开始前明确当前模式：

```plaintext
这次是 Explorer Mode，允许 Agent 自主推进。
```

或者：

```plaintext
这次是 Designer Mode，Agent 必须小步等待确认。
```

真正危险的是模式未声明：我以为自己在共同设计，Agent 却按快速交付在跑。

## 8. 最终理解

Agent 越强，越不能把开发者挤成旁观者。

我真正要训练的能力不是“让 Agent 一次性写出更多代码”，而是：

```plaintext
定义问题；
切出薄片；
设定边界；
控制 diff；
验证结果；
反思设计；
决定下一步。
```

Agent 可以成为高效的施工队，但它不应该替我决定建筑的形状。

我的目标不是成为一个等待 Agent 交付的项目主管，而是成为一个能够设计 Agent 工作流、系统边界和项目演化节奏的开发者。

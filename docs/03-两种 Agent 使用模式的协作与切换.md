# 两种 Agent 使用模式的协作与切换

> 索引：[[00-Agent 协作（索引）]]  
> 前置：[[01-像设计师一样使用 Agent]] ｜ [[02-像项目主管一样使用 Agent]]

## 0. 核心判断

“像设计师一样使用 Agent”和“像项目主管一样使用 Agent”不是互相冲突的两种理念，而是同一个项目在不同阶段需要的两种工作模式。

设计师模式解决的是：

```plaintext
我如何参与系统形成；
我如何控制边界；
我如何训练自己的工程判断；
我如何让项目一步步变得可理解、可维护。
```

项目主管模式解决的是：

```plaintext
我如何快速得到一个能跑的项目标本；
我如何利用 Agent 的自主执行力；
我如何在短时间内验证想法；
我如何让 Agent 自主推进但不失控。
```

它们真正的关系不是二选一，而是协作：

```plaintext
项目主管模式负责快速扩张，生成可观察对象；
设计师模式负责重新接管，定义系统边界；
加固模式负责工程化，让项目可靠演化。
```

因此，重点不是“用哪一种模式”，而是：

```plaintext
在同一个项目中，什么时候该快；
什么时候该慢；
什么时候该放权；
什么时候该收束；
什么时候该重新设计。
```

## 1. 问题：同一个项目里，开发者会反复切换身份

一个项目从想法到长期维护，通常不会只经历一种状态。

刚开始时，我可能只是有一个模糊想法：

```plaintext
我想做一个 agent runtime；
我想做一个任务管理工具；
我想做一个小型 SDK 示例；
我想做一个可用的原型。
```

这个阶段如果一开始就进入设计师模式，可能会过早陷入抽象：

```plaintext
概念还没验证，就开始设计目录；
核心闭环还没跑通，就开始设计插件系统；
用户路径还没走通，就开始定义复杂状态机；
项目价值还不确定，就开始追求长期架构。
```

但如果一直使用项目主管模式，又会出现另一个问题：

```plaintext
Agent 越写越多；
功能越来越完整；
目录越来越复杂；
我越来越看不懂；
最后项目虽然能跑，但已经不属于我。
```

所以同一个项目里，我需要在不同阶段扮演不同角色。

有时我是项目主管：

```plaintext
我设定目标和停止线，让 Agent 快速搭建可运行 demo。
```

有时我是设计师：

```plaintext
我重新定义边界、状态、接口、模块和验证方式。
```

有时我是审查者：

```plaintext
我让 Agent 停止写代码，解释现有系统。
```

有时我是加固者：

```plaintext
我把能跑的项目变成可靠、可维护、可演化的工程。
```

真正成熟的 Agent 工作流，不是固定一种姿态，而是有意识地切换姿态。

## 2. 职责边界

### 2.1 项目主管模式：让项目快速出现

项目主管模式的目标是快速得到一个完整、可运行、可观察的项目标本。

它允许 Agent 自主推进，但要求 Agent 留下：

```plaintext
目标；
假设；
决策；
阶段汇报；
验收方式；
临时方案；
后续重构建议。
```

项目主管模式关注的是：

```plaintext
这个东西能不能跑；
核心路径能不能走通；
需求之间是否冲突；
技术栈是否基本合适；
复杂度会集中在哪里；
后续是否值得继续投入。
```

它的关键词是：

```plaintext
快速、完整、可运行、可观察、可收束。
```

### 2.2 设计师模式：让项目重新属于我

设计师模式的目标不是快速堆功能，而是让我参与项目的形成过程。

它要求 Agent 小步推进：

```plaintext
先设计，再实现；
先确认边界，再改代码；
每次只做一个 checkpoint；
每轮改动保持可审核；
每一步都能运行和验证。
```

设计师模式关注的是：

```plaintext
系统语义是否清楚；
模块边界是否合理；
状态变化是否可解释；
接口是否会绑死未来；
抽象是否过早或过重；
我是否真的理解这个项目。
```

它的关键词是：

```plaintext
边界、语义、薄片、确认、可审核、可维护。
```

### 2.3 Review Mode：两种模式之间的缓冲层

项目主管模式不能直接跳到设计师模式。

因为快速生成的项目里会混杂很多东西：

```plaintext
真实核心；
临时方案；
Agent 的假设；
过度设计；
缺失边界；
无意中引入的技术债；
为了跑通 demo 而写下的捷径。
```

如果我不先审查，直接开始设计，很容易把 demo 里的偶然结构误认为正式架构。

所以两种模式之间需要一个 Review Mode。

Review Mode 的目标是停止新增功能，重新理解当前系统：

```plaintext
当前核心闭环是什么；
哪些模块可以保留；
哪些只是 demo 临时方案；
哪些抽象应该删除；
哪些边界需要重画；
哪些文件最值得人工审查；
下一轮应该进入设计、加固，还是丢弃原型。
```

Review Mode 是切换的关口。

它把项目从“Agent 生成物”重新变成“我可以理解的系统对象”。

## 3. 不要混用模式

两种模式可以在同一个项目中协作，但不能在同一个 session 里混用得太模糊。

最危险的情况是：

```plaintext
我心里想要设计师模式；
我希望参与每个边界；
但 prompt 写成了项目主管模式；
于是 Agent 自主实现了所有 feature。
```

或者：

```plaintext
我心里想要项目主管模式；
我需要快速拿到 demo；
但我又不断打断 Agent 的每个小实现；
导致原型迟迟跑不起来。
```

模式混用会导致预期错位：

```plaintext
我以为 Agent 会等我确认；
Agent 以为自己被授权自主推进。

我以为这只是 demo；
Agent 把它写成了伪 production 架构。

我以为这一步要快速验证；
Agent 却陷入细节优化和过度工程化。
```

所以每个 session 开始前，应该先声明当前模式：

```plaintext
本 session 使用 Project Manager Mode。
目标是快速生成可运行 demo。
Agent 可以自主推进，但必须记录假设、决策和停止线。
```

或者：

```plaintext
本 session 使用 Designer Mode。
目标是小步构建长期项目。
Agent 必须先给设计卡片，确认后才能实现。
```

模式声明不是形式主义，而是权限边界。

它告诉 Agent：

```plaintext
你这次有多大自主权；
你是否可以自行判断下一步；
你是否需要等我确认；
什么情况下必须停止；
本轮产物应该如何被评价。
```

## 4. 推荐生命周期：模式如何切换

一个比较稳定的项目节奏可以是：

```plaintext
0. Intent：产生想法
1. Project Manager Mode：快速生成原型
2. Review Mode：停止新增功能，审查原型
3. Designer Mode：重画边界，小步重构
4. Harden Mode：补测试、日志、错误处理和文档
5. Project Manager Mode：局部快速扩展新功能
6. Review Mode：审查新功能带来的复杂度
7. Designer Mode：把新功能纳入稳定架构
```

注意：项目主管模式不只出现在项目最开始。

在长期项目中，也可能有局部快速模式：

```plaintext
我要快速试一个插件机制；
我要快速试一个新 UI；
我要快速试一种存储后端；
我要快速试一个 agent task flow；
我要快速验证某个技术栈是否可用。
```

这时可以在项目的某个分支或实验目录中临时进入项目主管模式。

但实验结束后，必须回到 Review Mode 和 Designer Mode。

否则实验代码会直接污染主架构。

## 5. 切换门：什么时候切换模式

### 5.1 从项目主管模式切到 Review Mode

项目主管模式应该在核心闭环跑通后立刻停止，而不是继续扩张。

出现以下信号，就应该切到 Review Mode：

```plaintext
项目已经可以本地启动；
核心用户路径可以完整走通；
README 能说明运行方式；
主要功能已经覆盖 demo 目标；
Agent 开始建议继续加高级功能；
目录结构开始膨胀；
我已经无法在脑中描述主要模块关系。
```

这时不要继续说：

```plaintext
顺便加一下权限；
顺便加一下部署；
顺便加一下插件；
顺便加一下 UI 美化；
顺便把架构优化一下。
```

正确动作是：

```plaintext
停止新增功能；
输出项目地图；
列出假设和决策；
标记临时方案；
给出人工审查入口；
进入 Review Mode。
```

项目主管模式的成功标志不是“功能越来越多”，而是“原型已经足够让我看清项目形态”。

### 5.2 从 Review Mode 切到设计师模式

Review Mode 的目标是重新理解项目。

当我已经回答这些问题后，就可以进入设计师模式：

```plaintext
核心闭环是什么；
哪些功能是真需求；
哪些功能可以删除；
哪些模块可以保留；
哪些模块只是 demo 临时实现；
哪些边界必须重画；
下一轮最小 checkpoint 是什么。
```

进入设计师模式时，不能要求 Agent “继续完善整个项目”。

应该重新切成薄片：

```plaintext
只重构 task 状态模型；
只整理配置加载边界；
只把存储层从业务逻辑中拆出来；
只给核心路径补测试；
只把 CLI 的输入输出稳定下来。
```

设计师模式的目标不是在 demo 上继续堆功能，而是把 demo 中被验证过的核心抽出来，变成可维护的系统结构。

### 5.3 从设计师模式切回项目主管模式

设计师模式不是永远慢下去。

在长期项目中，某些新方向仍然适合快速探索。

比如：

```plaintext
我已经有稳定 core，现在想试一个新 feature；
我不确定某个第三方库是否适合；
我想比较两种 UI 交互；
我想验证一个 agent workflow 是否可行；
我想快速生成一个 examples/ 目录展示用法。
```

此时可以局部切回项目主管模式。

但要满足三个条件：

```plaintext
实验范围隔离；
停止线明确；
实验结束必须 Review。
```

例如：

```plaintext
本 session 使用 Project Manager Mode，但只允许在 examples/experimental-plugin/ 下工作。
不要修改 core 模块。
目标是验证插件调用链是否可行。
核心路径跑通后停止，并输出 Review 报告。
```

这能避免快速模式破坏已有架构。

## 6. 协作原则

### 6.1 快速模式产出材料，设计模式提炼结构

项目主管模式产出的不是最终架构，而是材料：

```plaintext
可运行路径；
目录形态；
技术假设；
功能冲突；
复杂度分布；
用户路径；
失败点；
临时代码。
```

设计师模式要做的是从这些材料中提炼结构：

```plaintext
真正的核心对象是什么；
状态机应该如何定义；
哪些模块应该合并；
哪些模块应该拆开；
哪些接口需要稳定；
哪些功能应该延后；
哪些代码应该删除。
```

不要把快速模式的结果直接当成设计结论。

快速模式回答的是：

```plaintext
这个方向大概能不能跑？
```

设计师模式回答的是：

```plaintext
这个系统应该如何长？
```

### 6.2 Agent 可以在快速模式中做决定，但必须在设计模式中接受审查

在项目主管模式里，Agent 可以做最小合理假设。

但这些假设不是最终决定。

它们需要在 Review Mode 中被重新审查：

```plaintext
这个假设是否影响架构边界？
这个技术选择是否只是为了 demo 方便？
这个抽象是否是 Agent 自己脑补的？
这个依赖是否真的值得保留？
这个模块是否应该被删除？
```

Agent 的自主权是阶段性的。

它可以在快速探索时提高速度，但不能在长期设计中越过开发者的判断。

### 6.3 每次切换模式，都要重写任务协议

不要假设 Agent 会自动理解当前模式已经变了。

从项目主管模式切到设计师模式时，要明确说：

```plaintext
现在停止自主扩展。
接下来进入 Designer Mode。
你必须先输出设计卡片，确认后才能改代码。
每轮只做一个 checkpoint。
```

从设计师模式切到项目主管模式时，也要明确说：

```plaintext
现在进入局部 Project Manager Mode。
你可以自主推进，但只限于指定实验范围。
核心闭环跑通后停止，不要修改稳定 core。
```

模式切换必须显式化。

否则 Agent 会沿用上一个阶段的权限。

## 7. 工作卡片：Mode Card

为了避免混乱，可以在每次使用 Agent 前先写一张模式状态卡。

```markdown
# Mode Card

## 当前模式
Project Manager Mode / Review Mode / Designer Mode / Harden Mode

## 本 session 目标
写清楚这次只解决什么问题。

## Agent 权限
- 是否可以自主判断下一步？
- 是否可以直接改代码？
- 是否必须先给方案？
- 是否需要等待我确认？

## 工作范围
允许修改哪些目录 / 文件？
禁止修改哪些目录 / 文件？

## 停止线
什么情况下必须停止新增功能？

## 输出物
本轮结束时必须给出什么？
```

这张卡片的价值在于：它把“我想怎么协作”变成了明确协议。

## 8. 参考 Prompt

### 8.1 项目开始时选择模式

```markdown
我们要在同一个项目中交替使用几种 Agent 协作模式。

开始前请先确认当前 session 的模式，不要直接写代码。

请根据下面格式输出 Mode Card：

# Mode Card

## 当前模式
从以下模式中选择一个：
- Project Manager Mode：快速搭建，可自主推进
- Review Mode：停止新增功能，审查现有系统
- Designer Mode：小步设计，确认后实现
- Harden Mode：工程加固，补测试、日志、文档、错误处理

## 适用理由
为什么当前应该使用这个模式？

## 本 session 目标
本轮只解决什么？

## Agent 权限
你是否可以直接改代码？
你是否需要等待我确认？
你是否可以自主判断下一步？

## 工作范围
允许修改哪些文件或目录？
禁止修改哪些文件或目录？

## 停止线
什么时候必须停止？

## 输出物
本轮结束时必须交付什么？

输出 Mode Card 后等待我确认。
```

### 8.2 从项目主管模式切到 Review Mode

```markdown
现在停止新增功能。

我们从 Project Manager Mode 切换到 Review Mode。

请不要继续写代码。

请基于当前项目输出：

1. 当前核心闭环是什么；
2. 已实现功能列表；
3. 哪些功能超出了最初 demo 目标；
4. 当前目录结构说明；
5. 关键假设列表；
6. 关键设计决策列表；
7. 临时方案和 TODO(demo)；
8. 最值得人工审查的 5 个文件；
9. 当前最危险的技术债；
10. 是否建议进入 Designer Mode、Harden Mode，还是丢弃原型。

注意：
你的任务是帮助我重新理解项目，而不是继续完成项目。
```

### 8.3 从 Review Mode 切到 Designer Mode

```markdown
现在从 Review Mode 切换到 Designer Mode。

请不要直接改代码。

基于刚才的审查结果，请设计下一轮最小 checkpoint。

请用设计卡片格式输出：

# 本轮目标
# 为什么选择这个 checkpoint
# 本轮不做什么
# 核心对象
# 状态变化
# 文件影响
# 需要删除或收敛的 demo 代码
# 风险
# 验证方式
# 等待我确认的问题

确认前不要写代码。
```

### 8.4 从 Designer Mode 切到局部项目主管模式

```markdown
现在从 Designer Mode 临时切换到局部 Project Manager Mode。

目标：快速验证一个实验方向，不影响稳定 core。

工作规则：

1. 你可以自主推进，不需要每一步等待我确认。
2. 只允许在以下范围内工作：
   - examples/
   - experiments/
   - docs/experiments/
3. 不要修改 core 模块、稳定 API、数据库 schema 或公共接口。
4. 如果必须修改稳定区域，先停下并说明原因。
5. 所有假设记录到 ASSUMPTIONS.md。
6. 所有关键决策记录到 DECISIONS.md。
7. 核心实验路径跑通后停止新增功能。
8. 最终输出实验结果、验证方式、是否值得纳入主架构。

这个实验产物不是正式架构。
```

### 8.5 从 Designer Mode 切到 Harden Mode

```markdown
现在从 Designer Mode 切换到 Harden Mode。

目标：不新增核心功能，只提高当前系统的可靠性和可维护性。

请先输出加固计划，不要直接改代码：

1. 当前最需要补的测试是什么？
2. 哪些错误路径没有处理？
3. 哪些日志或调试信息缺失？
4. README / docs 哪些地方需要补充？
5. CI 或格式化检查是否需要补？
6. 哪些地方需要减少临时代码？
7. 请给出 5 个最小加固 checkpoint。

确认后，每次只实现一个 checkpoint。
```

### 8.6 模式混乱时重置

```markdown
当前协作模式已经混乱。

请停止当前动作，不要继续写代码。

请先帮我整理：

1. 当前实际处于什么模式？
2. 我原本可能想要什么模式？
3. 现在的改动是否超出了当前目标？
4. 哪些内容应该保留？
5. 哪些内容应该撤回或推迟？
6. 接下来应该切换到哪个模式？
7. 请重新输出一张 Mode Card。

在我确认 Mode Card 前，不要继续实现。
```

## 9. 危险信号：说明需要切换模式

### 9.1 需要从项目主管模式切到 Review Mode 的信号

```plaintext
demo 已经能跑；
Agent 开始建议继续加高级功能；
功能增长快于理解增长；
目录结构开始膨胀；
我已经不知道哪些是核心；
README 很完整但我不理解实现；
临时代码没有被标记；
假设和决策没有被记录。
```

### 9.2 需要从 Review Mode 切到设计师模式的信号

```plaintext
我已经理解核心闭环；
我知道哪些代码可以保留；
我知道哪些功能应该删除；
我已经找到了下一个最小重构点；
我需要重新定义状态机、接口或模块边界。
```

### 9.3 需要从设计师模式切到项目主管模式的信号

```plaintext
我遇到一个尚未验证的新方向；
继续精细设计会过早；
我需要比较多个技术方案；
我需要快速生成一个例子；
我只想在隔离范围内试错。
```

### 9.4 需要从设计师模式切到 Harden Mode 的信号

```plaintext
核心边界已经稳定；
功能已经够用；
继续新增功能收益不高；
错误处理薄弱；
测试缺失；
文档不足；
我希望这个项目进入可维护状态。
```

## 10. 最终理解

同一个项目中，Agent 协作不是一条直线，而是一组可切换的工作模式。

项目主管模式让我获得速度。

设计师模式让我获得理解。

Review Mode 让我重新接管。

Harden Mode 让我获得可靠性。

它们之间的关系可以概括为：

```plaintext
Project Manager Mode：让系统快速出现。
Review Mode：让系统重新可理解。
Designer Mode：让系统按照我的边界生长。
Harden Mode：让系统能够长期存在。
```

真正危险的不是使用快速模式，也不是使用慢速模式。

真正危险的是：

```plaintext
没有声明模式；
没有切换门；
没有停止线；
没有审查层；
没有重新接管项目的动作。
```

我不需要在每一刻都亲自参与所有实现细节，也不应该把所有判断都交给 Agent。

更好的方式是：

```plaintext
需要速度时，我给 Agent 执行自主权；
需要设计时，我收回边界定义权；
需要理解时，我停止新增功能；
需要长期维护时，我进入加固节奏。
```

Agent 是强大的执行系统，但项目的节奏、边界和解释权仍然应该由我掌握。

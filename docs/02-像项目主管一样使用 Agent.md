# 像项目主管一样使用 Agent

> 索引：[[00-Agent 协作（索引）]]  
> 前置：[[01-像设计师一样使用 Agent]] ｜ 汇总：[[03-两种 Agent 使用模式的协作与切换]]

## 0. 核心判断

开发者有时需要像设计师一样使用 Agent：小步确认、控制边界、参与每个抽象的形成。

但也有另一种场景：我不想一步步确认，我想让 Agent 在一个 session 里快速推进，自己判断下一步，尽快搭建出一个完整、可运行、可观察的项目。

这时我不是设计师，而更像项目主管。

项目主管不是把所有细节都亲自做完，也不是完全放任施工队乱建，而是负责：

- 明确目标
- 设定边界
- 分配权限
- 定义验收标准
- 控制风险
- 要求阶段汇报
- 在关键节点接管判断

也就是说，项目主管式使用 Agent 的重点不是“每一步都参与”，而是“让 Agent 自主推进，但始终处在可验收、可追踪、可收束的框架里”。

```plaintext
你：我要一个能跑的项目 demo，目标和边界如下。
agent：我会按阶段推进，记录假设、决策、临时方案，并在完成后给出验收方式。
你：我不逐步确认实现细节，但我要能接管、审查、重构。
```

## 1. 适用场景

项目主管模式适合这些情况：

```plaintext
我想快速验证一个想法是否成立；
我需要一个能跑的 demo；
我暂时不追求最终架构；
我想让 Agent 自主完成大部分执行；
我希望在一个 session 内看到项目完整形态；
我需要用完整原型反推真实复杂度。
```

它不适合这些情况：

```plaintext
这是一个长期项目的核心架构；
我需要训练自己的系统设计判断；
每个抽象都很重要；
代码必须从一开始就高度可维护；
我不能接受 Agent 自行做技术假设。
```

如果目标是学习、设计、沉淀架构，应使用“像设计师一样使用 Agent”。

如果目标是快速形成原型、跑通闭环、获得工程标本，可以使用“像项目主管一样使用 Agent”。

## 2. 问题：放权不等于失控

很多人用 Agent 快速开发时，最大的问题不是 Agent 不够强，而是授权方式太模糊。

错误的项目主管模式是：

```plaintext
你：帮我做一个完整项目。
agent：好的，我实现了十几个模块，引入了多个依赖，还加了很多我觉得必要的功能。
你：代码太多，我不知道怎么审，也不知道哪些是核心。
```

这不是高效，而是失控。

真正的项目主管模式应该是：

```plaintext
你：你可以自主推进，但必须遵守目标、边界、记录、验收、停止线。
agent：我会快速实现，但所有假设、决策、临时代码和后续风险都会留下材料。
你：我不逐行参与，但我能在结束后接管项目。
```

项目主管式使用 Agent 的关键，是让 Agent 拥有执行自主权，但不拥有最终解释权。

## 3. 工作机制：七个控制点

### 3.1 Goal Lock：目标锁

目标锁：先定义本次 session 的唯一目标。

快速模式最怕目标过宽。目标过宽时，Agent 会倾向于把所有可能相关的功能都塞进去。

目标应该写成一个明确的端到端闭环：

```plaintext
本次目标不是完成最终产品，而是完成一个可运行 demo。

demo 必须覆盖：
1. 用户可以完成核心操作；
2. 系统能保存或展示核心状态；
3. 出错时有基本提示；
4. 我可以通过 README 运行和验证它。
```

目标锁的作用是告诉 Agent：

```plaintext
你可以自主推进，但只能朝这个目标推进。
```

### 3.2 Scope Guard：范围护栏

范围护栏：允许 Agent 自主判断下一步，但不允许无限扩展。

项目主管模式不是每一步都确认，但必须提前写清楚什么不做。

例如：

```plaintext
本轮不做：
- 复杂权限系统
- 多租户
- 支付
- 云部署
- 高级缓存
- 微服务拆分
- 过度抽象的插件系统
- 大规模性能优化
```

范围护栏的意义是防止 Agent 把 demo 写成“假装成熟的复杂系统”。

快速原型可以粗糙，但不能膨胀。

### 3.3 Decision Log：决策日志

决策日志：Agent 可以做决定，但必须留下记录。

项目主管模式允许 Agent 在遇到小问题时做最小合理假设，不需要每次都询问我。

但所有关键假设和设计选择，都必须记录下来。

至少需要两个文件：

```plaintext
ASSUMPTIONS.md
记录需求不明确时 Agent 做了哪些假设。

DECISIONS.md
记录关键技术选择、替代方案、选择理由和潜在风险。
```

如果没有决策日志，Agent 生成的项目会变成黑箱。

有了决策日志，我才能知道：

```plaintext
它为什么这样设计；
哪些地方只是临时选择；
哪些地方以后需要重新评估；
哪些模块是 Agent 自己脑补的。
```

### 3.4 Progress Report：阶段汇报

阶段汇报：Agent 可以连续工作，但必须周期性报告。

快速模式不要求每一步都等我确认，但不能完全沉默地一路写到底。

我需要 Agent 在完成每个阶段后输出简短汇报：

```plaintext
# 阶段进展
完成了什么：
涉及文件：
如何运行：
如何验证：
当前假设：
当前风险：
下一阶段：
```

阶段汇报不是为了打断 Agent，而是为了让项目进度可观测。

如果没有阶段汇报，我就只能在最后面对一个庞大的结果。

项目主管模式要避免这种情况：

```plaintext
过程不可见；
决策不可见；
复杂度不可见；
风险不可见；
最后只能被动验收。
```

### 3.5 Acceptance Lock：验收锁

验收锁：在开始前定义什么叫“完成”。

快速模式最常见的问题是：Agent 一直继续加功能，直到上下文快耗尽。

所以必须提前定义完成条件。

例如：

```plaintext
当满足以下条件时停止新增功能：

1. 项目可以本地启动；
2. 核心用户路径可以走通；
3. README 说明了运行方式；
4. 至少有一种验证方式；
5. 已知问题被记录；
6. 后续重构建议被列出。
```

验收锁的作用是把 Agent 从“继续产出”拉回“完成交付”。

项目主管模式下，完成不是“所有想象中的功能都实现了”，而是“本轮目标已经形成可验收结果”。

### 3.6 Stop Line：停止线

停止线：一旦核心闭环跑通，禁止继续扩张。

快速模式不是一直快下去。

一旦 demo 跑通，就应该切换到 Review Mode，而不是继续追加功能。

停止线可以这样写：

```plaintext
如果核心闭环已经跑通，不要继续新增功能。
请停止实现，转为整理：
- 项目结构说明
- 已实现功能
- 未实现功能
- 已知问题
- 临时方案
- 后续重构计划
```

停止线是项目主管模式最重要的安全装置。

它防止 Agent 从“快速搭建”滑向“无限施工”。

### 3.7 Review Gate：审查门

审查门：快速实现之后，必须重新接管项目。

项目主管模式的后半段不是继续让 Agent 写，而是让它解释当前项目。

完成 demo 后，我应该要求 Agent 输出架构审查报告：

```plaintext
当前项目的核心闭环是什么？
哪些模块是稳定设计？
哪些模块只是 demo 临时实现？
哪些地方存在过度设计？
哪些地方边界不清？
哪些文件最需要人工审查？
如果要转为长期项目，第一轮重构应该怎么做？
```

Review Gate 的意义是：

```plaintext
Agent 负责快速生成标本；
我负责解剖标本；
之后再决定是否重构成长期项目。
```

## 4. 参考 Prompt

### 4.1 项目开始时使用

```markdown
你现在进入 Project Manager Mode：项目主管式快速搭建模式。

目标：在一个 session 中尽快搭建一个完整、可运行、可验证的项目 demo。

工作方式：

1. 你可以自主判断下一步，不需要每一步都等待我确认。
2. 优先完成端到端核心闭环，而不是追求完美架构。
3. 你可以做最小合理假设，但必须记录到 ASSUMPTIONS.md。
4. 所有关键设计选择必须记录到 DECISIONS.md。
5. 不要过度引入复杂框架、复杂抽象或不必要的基础设施。
6. 不要把 demo 写成 production-ready 系统。
7. 所有为了快速 demo 写下的临时方案，请用 TODO(demo) 或 FIXME 标记。
8. 每完成一个阶段，输出阶段汇报：
   - 完成了什么
   - 修改了哪些文件
   - 如何运行
   - 如何验证
   - 当前风险
   - 下一阶段计划
9. 当核心闭环跑通后，停止新增功能。
10. 最终必须输出：
   - 项目结构
   - 启动方式
   - 功能清单
   - 验证方式
   - 已知问题
   - 临时方案
   - 后续重构计划

注意：
这个 demo 不是最终架构。
你的目标是快速探索项目形态，为后续 Review Mode / Designer Mode / Harden Mode 提供材料。
```

### 4.2 需求不清楚但不想频繁打断时使用

```markdown
如果遇到需求不明确的地方，不要立即停下询问。

请按以下规则处理：

1. 对影响不大的细节，做最小合理假设，并记录到 ASSUMPTIONS.md。
2. 对影响架构边界的选择，暂停并向我提问。
3. 对安全、权限、数据删除、付费、外部服务调用等高风险事项，不要自行决定。
4. 如果你做了临时选择，必须在代码或文档中标记 TODO(demo)。

继续以完成核心闭环为优先。
```

### 4.3 Agent 开始过度扩张时使用

```markdown
停止扩张。

当前 session 的目标是可运行 demo，不是完整产品。

请立刻执行以下操作：

1. 停止新增功能。
2. 列出已经实现的功能。
3. 标记哪些功能超出了本轮目标。
4. 给出可以删除或推迟的部分。
5. 回到核心闭环，优先保证它可运行、可验证。
```

### 4.4 项目已经跑通、需要收束时使用

```markdown
核心闭环已经跑通。

现在停止新增功能，进入收束阶段。

请输出：

1. 项目结构说明
2. 启动方式
3. 核心用户路径
4. 已实现功能
5. 未实现功能
6. 已知问题
7. 临时方案
8. 后续重构建议
9. 最值得人工审查的 5 个文件

不要继续写新代码。
```

### 4.5 快速模式结束后的 Review Prompt

```markdown
现在进入 Review Mode。

请基于当前项目输出一份架构审查报告：

1. 当前项目的核心闭环是什么？
2. 当前项目最重要的模块有哪些？
3. 哪些设计可以保留？
4. 哪些设计只是 demo 临时方案？
5. 哪些地方存在过度设计？
6. 哪些地方边界不清？
7. 哪些依赖可以移除或替换？
8. 哪些文件最需要我人工审查？
9. 当前最危险的技术债是什么？
10. 如果要转为长期项目，请给出 5 个最小重构 checkpoint。

注意：
不要直接改代码。
先帮我重新理解项目。
```

## 5. 推荐开发节奏

0. Intent：本次 session 的目标是什么
1. Outcome：什么结果算完成
2. Scope：哪些事情必须做，哪些事情明确不做
3. Assumption：需求不清时允许做哪些假设
4. Plan：Agent 自主推进的大致阶段
5. Execution：Agent 连续实现
6. Report：阶段性汇报
7. Acceptance：按验收标准判断是否完成
8. Review：停止新增功能，审查结构
9. Handoff：决定进入设计模式、加固模式，或直接丢弃原型

注意第 8 步很重要。

项目主管不是一直催促施工，而是在项目形成可运行形态后，立刻组织审查和接管。

## 6. 工作卡片：项目进展卡片

```markdown
请用项目进展卡片格式汇报：

# 当前阶段
# 已完成
# 涉及文件
# 运行方式
# 验证方式
# 当前假设
# 当前风险
# 下一步
# 是否接近停止线
```

这个卡片的作用不是让我逐行审核，而是让我保持对项目状态的可观测性。

## 7. 与其他模式的关系

“像设计师一样使用 Agent”和“像项目主管一样使用 Agent”不是互相否定，而是两种不同模式。

设计师模式关注：

```plaintext
小步推进；
参与每个设计判断；
控制每个抽象边界；
每一步都可审核；
长期可维护。
```

项目主管模式关注：

```plaintext
快速形成完整 demo；
允许 Agent 自主推进；
通过日志和汇报保持可观测；
通过停止线避免失控；
通过 Review Mode 重新接管。
```

两者的关系可以理解为：

```plaintext
项目主管模式：先让 Agent 快速生成工程标本。
设计师模式：再由我重新定义边界和系统语义。
加固模式：最后补测试、文档、错误处理和长期维护能力。
```

不要在一个 session 里混用两种模式。

如果当前是设计师模式，就不要让 Agent 自主实现一切。

如果当前是项目主管模式，就不要期待自己参与每个实现细节。

真正重要的是在开始前声明模式，并在结束时切换模式。

## 8. 危险信号

项目主管模式中，一旦出现下面的情况，应该立刻停止新增功能：

```plaintext
Agent 开始引入大型框架；
目录结构突然膨胀；
核心闭环还没跑通，却开始做高级功能；
每个模块都看似合理，但整体意图变模糊；
README 写得很漂亮，但项目跑不起来；
出现大量我无法解释的抽象；
没有 ASSUMPTIONS.md 或 DECISIONS.md；
Agent 开始“顺手”重构无关代码；
功能越来越多，但验收标准越来越模糊。
```

这些信号说明项目主管模式已经从快速搭建变成复杂度制造。

此时应该立刻切换到收束：

```plaintext
停止新增功能；
输出项目地图；
列出假设和决策；
标记临时方案；
给出重构 checkpoint。
```

## 9. 最终理解

像项目主管一样使用 Agent，不是让 Agent 替我做项目负责人。

它的意义是：

```plaintext
我定义目标；
我设定边界；
我给出验收标准；
Agent 自主推进；
Agent 留下决策记录；
我在停止线后接管判断。
```

项目主管模式承认一个事实：

```plaintext
有些阶段，速度比精致更重要；
有些问题，只有先跑起来才能看清楚；
有些设计，需要先看到一个粗糙标本，才能反推真正的边界。
```

但它也必须坚持另一个原则：

```plaintext
Agent 可以拥有执行自主权；
不能拥有最终解释权。
```

快速搭建的最终目的不是得到一个完美项目，而是得到一个可以被观察、审查、重构的工程对象。

我不需要在每一步都站在施工现场，但我必须知道：

```plaintext
项目为什么这样长；
哪些地方是临时的；
哪些地方可以保留；
什么时候该停止；
之后该如何接管。
```
